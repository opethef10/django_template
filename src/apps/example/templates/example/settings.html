    {% translate "Development Settings" %}

    {% translate "When you're working on your local machine, you need settings that make development easy and fast. The development settings are designed to give you as much information as possible when something goes wrong, and to avoid accidentally affecting real users." %}

    {% translate "DEBUG = True" %}

    {% translate "When DEBUG is True, Django shows detailed error pages whenever something goes wrong. Instead of just seeing 'Something went wrong', you'll see exactly which line of code caused the problem, what the error is, and the full stack trace. This makes fixing bugs much easier. In production, this would be a security risk because attackers could learn about your server setup, but locally it's incredibly helpful." %}

    {% translate "ALLOWED_HOSTS = ['*']" %}

    {% translate "ALLOWED_HOSTS is a security feature that tells Django which domain names are allowed to access your site. When developing locally, you access your site through various addresses like localhost, 127.0.0.1, or maybe a custom hostname. Instead of configuring this every time, we use '*' which means 'allow everything'. This is fine for local development but would be dangerous in production." %}

    {% translate "Debug Toolbar" %}

    {% translate "The debug toolbar is a powerful tool that appears on the right side of your pages when you're developing. It shows you lots of useful information: how long the page took to load, what database queries were made (and how long each took), what template variables are available, and more. This helps you understand how your code is performing and what's happening behind the scenes. It's only available in development because it reveals too much information about your server to be safe in production." %}

    {% translate "Console Email Backend" %}

    {% translate "When your code sends an email (like when someone signs up and needs to confirm their email), Django needs to know how to deliver it. In development, we don't want to actually send emails - we want to see what would have been sent. The console email backend prints all emails to your terminal instead of sending them. This way, you can test email-related features without worrying about accidentally emailing real users or needing to set up a real email server." %}

    {% translate "Dummy Cache" %}

    {% translate "Caching is a technique where Django stores frequently-used data in memory so it can be retrieved faster. For example, instead of querying the database every time someone visits a page, Django can save the result and serve it instantly. In production, you typically use a cache server like Redis or Memcached. However, setting up a cache server adds complexity to your local development environment. The dummy cache pretends to cache things but doesn't actually store anything - this keeps your development setup simple while still allowing code that uses caching to work without errors." %}

    {% translate "Production Settings" %}

    {% translate "When you deploy your website to a live server, you need completely different settings. Production is about security, performance, and reliability. You don't need helpful error messages anymore - instead, you need to protect your users' data and keep your server running smoothly." %}

    {% translate "DEBUG = False" %}

    {% translate "When DEBUG is False, Django hides all error details from users. Instead of seeing a helpful error page with the exact problem, users just see a generic 'Something went wrong' message. This might seem unhelpful, but it's actually important for security. If a hacker tries to break into your site and sees detailed error messages, they can learn about your server setup, what databases you're using, and where your code might have weaknesses. By hiding this information, you make it much harder for attackers to find vulnerabilities." %}

    {% translate "ALLOWED_HOSTS" %}

    {% translate "In production, you should only allow access from your actual domain name(s). This prevents a type of attack called 'host header injection' where an attacker tries to use your server for malicious purposes by sending requests with fake host names. By specifying exactly which domains are allowed, you ensure that only legitimate traffic reaches your application." %}

    {% translate "File Logging" %}

    {% translate "When something goes wrong in production, you can't just look at the error page on your screen - users are seeing the site, not you. That's where logging comes in. Every time something important happens (an error, a user signing up, a payment being processed), Django writes a record to a log file. This log file lives at /var/www/proj.log on your server. When something breaks, you can check this file to see what happened. Logging is essential for troubleshooting problems that you can't see directly." %}

    {% translate "Static and Media Files" %}

    {% translate "Static files are things like CSS, JavaScript, and images that don't change - they're the same for all users. Media files are uploads like profile pictures or documents that users send to your site. In development, Django serves these files directly. In production, it's much more efficient to let your web server (Nginx) serve these files directly, bypassing Django entirely. This makes your site much faster because static files can be cached by browsers and served from CDNs. We configure Django to look for these files in /var/www/static and /var/www/media." %}

    {% translate "Real SMTP Email" %}

    {% translate "In production, when users sign up, request password resets, or contact you through forms, they need to actually receive those emails. This requires a real email server (SMTP). We use Gmail's SMTP server as a simple option, but you can use any email provider. The email is only enabled if DJANGO_EMAIL_ENABLED=True - this allows you to disable email completely if needed without changing code." %}

    {% translate "Test Settings" %}

    {% translate "When you run automated tests, you need settings that make tests run as fast as possible. Tests that take minutes to run get annoying and people stop running them. The test settings are optimized for speed, not security or accuracy." %}

    {% translate "MD5 Password Hasher" %}

    {% translate "Password hashing is the process of turning a password into a scrambled version that can't be reversed. When users create accounts, their passwords are hashed and the hash is stored. When they log in, their password is hashed again and compared to the stored hash. This is secure because even if someone steals the database, they can't see the actual passwords. However, secure hashing algorithms like bcrypt are designed to be slow - that's intentional because it makes brute force attacks harder. The problem is, in tests, you're constantly creating users and checking passwords. Using MD5 (which is fast but not very secure) makes tests run much faster. Since tests run in an isolated environment, this security trade-off is acceptable." %}

    {% translate "Disable Migrations" %}

    {% translate "Normally, before running tests, Django creates a fresh test database and runs all your migrations to set up the tables. This can take a long time if you have many migrations. The test settings use a trick: instead of running migrations, Django creates the tables directly from your model definitions. This skips a lot of the overhead and makes test setup much faster." %}

    {% translate "Disable Logging" %}

    {% translate "Every time Django logs something, there's overhead involved - formatting the message, checking log levels, and actually writing it somewhere. During tests, this logging output isn't useful (you're not watching the test run for errors), so we disable it entirely to speed things up." %}
